Resumo Executivo

O Transparência Social é uma plataforma de prestação de contas para projetos sociais. A Organização responsável pelo projeto cria relatórios periódicos, preenche os dados de execução e financeiros (itens de orçamento, gastos, remanejamentos, comprovantes e extratos bancários), e os envia para análise.

A análise e aprovação segue uma regra dinâmica: se o projeto possui um Consultor ativo vinculado, é ele quem analisa e decide. Se não há Consultor, o Investidor (financiador) assume esse papel diretamente.

Caso o relatório seja devolvido, a Organização recebe o comentário do analista, corrige e reenvia — esse ciclo se repete até a aprovação final. Uma vez aprovado, o relatório é travado e todos os dados financeiros são consolidados automaticamente na aba Financeiro do projeto.

O sistema possui três perfis com papéis claros: a Organização opera (cria, preenche, envia); o Consultor, quando presente, analisa e aprova; o Investidor aprova diretamente quando não há Consultor, e sempre mantém visibilidade total.

Fluxo Geral do Sistema

O ciclo de vida de um relatório no Transparência Social segue uma jornada com quatro status possíveis e papéis bem definidos em cada etapa.

Status do Relatório

Rascunho → Enviado → Aprovado
Enviado → Devolvido → Rascunho (ciclo de ajustes)

Transições Implementadas

RASCUNHO → ENVIADO (Organização envia para análise)
ENVIADO → APROVADO (Consultor ou Investidor aprova)
ENVIADO → DEVOLVIDO (Consultor ou Investidor devolve)
DEVOLVIDO → ENVIADO (Organização ajusta e reenvia)
APROVADO → nenhuma transição (status final, travado)

Jornada Completa

A Organização cria um novo relatório para o projeto, que nasce com status Rascunho. Neste estágio, a Organização preenche todas as seções: dados gerais de execução, itens financeiros (orçamento planejado, gastos por período, remanejamentos), comprovantes de despesas (notas fiscais e recibos) e extratos bancários.

Ao concluir o preenchimento, a Organização envia o relatório para análise. O status muda para Enviado e o relatório fica bloqueado para edição. Neste ponto, o sistema verifica automaticamente se o projeto possui um Consultor ativo vinculado.

Se houver Consultor, ele é quem receberá o relatório para análise e terá autoridade para aprovar ou devolver. Caso contrário, o Investidor assume a análise. O analista responsável tem duas opções: aprovar o relatório (status final: Aprovado) ou devolver para ajustes (status: Devolvido), obrigatoriamente incluindo um comentário com o que precisa ser corrigido.

Se devolvido, a Organização recebe o relatório de volta, faz os ajustes necessários e reenvia. Esse ciclo se repete quantas vezes for necessário até a aprovação. Ao ser aprovado, o relatório é permanentemente travado e todos os dados financeiros ficam disponíveis na visão consolidada do projeto.

Módulo Financeiro

Os dados financeiros são preenchidos dentro de cada relatório, em tabelas normalizadas separadas por categoria: itens financeiros (seção 12), remanejamentos (seção 13), comprovantes de despesas com upload de arquivos (seção 14) e extratos bancários (seção 15).

A aba Financeiro do Projeto exibe uma visão consolidada automaticamente, agregando os dados financeiros de todos os relatórios do projeto. Essa visão inclui totais de orçamento planejado, gastos realizados, valor total de comprovantes e contagem de extratos bancários.

Fluxo por Perfil

Perfil Principal

Organização

A Organização é o perfil operacional do sistema. É sempre ela quem inicia e conduz o preenchimento dos relatórios.

  • Cria novos relatórios dentro dos projetos que administra
  • Preenche todas as seções: dados de execução, itens financeiros, comprovantes e extratos
  • Envia o relatório para análise quando o preenchimento está completo (exige ao menos 1 comprovante)
  • Edita relatórios em Rascunho e também relatórios Devolvidos (após devolução pelo analista)
  • Reabre relatórios devolvidos para edição antes de reenviar
  • Não participa da decisão de aprovação — essa responsabilidade é do Consultor ou do Investidor
Quando o relatório é devolvido: A Organização recebe o relatório com status "Devolvido" junto ao comentário do analista explicando os ajustes. Pode então editar o conteúdo, corrigir os pontos indicados e reenviar para nova análise. O ciclo se repete quantas vezes for necessário.
Perfil Especializado

Consultor

O Consultor é vinculado a projetos específicos. Nem todo projeto possui Consultor — apenas aqueles que têm um consultor ativo na tabela de vínculo do projeto.

  • Entra no fluxo quando está vinculado e marcado como ativo em um projeto específico
  • Analisa os relatórios enviados pela Organização que estejam com status Enviado
  • Aprova ou devolve os relatórios — sua decisão é final
  • Substitui completamente o Investidor na função de análise — quando existe Consultor, o Investidor não pode aprovar ou devolver
  • Não cria, não edita, não envia relatórios — seu papel é exclusivamente de análise e decisão
Quando não participa: Se o Consultor não está vinculado ao projeto ou não está marcado como ativo, ele não tem papel nenhum no fluxo de aprovação daquele projeto. Nesse cenário, o Investidor assume diretamente.
Perfil Financiador

Investidor

O Investidor é o financiador do projeto. Seu nível de envolvimento operacional na aprovação depende da existência de um Consultor.

  • Analisa e aprova diretamente quando o projeto não possui Consultor ativo vinculado
  • Não participa da aprovação quando o projeto possui Consultor — a decisão é inteiramente do Consultor
  • Tem visibilidade total sobre os projetos, relatórios e dados financeiros consolidados
  • Pode criar relatórios e editar rascunhos iniciais, quando necessário
  • Pode enviar relatórios em status Rascunho
Regra principal: O Investidor atua diretamente na aprovação apenas nos projetos sem Consultor. Nos projetos com Consultor, ele mantém visibilidade completa, mas a decisão de aprovação é delegada ao Consultor.

Regras de Negócio Implementadas

Todas as regras abaixo estão implementadas e verificadas diretamente no código-fonte do sistema.

1
A Organização é quem cria e preenche o relatório. O Investidor também pode criar, mas a Organização é o perfil operacional principal. report.actions.ts
2
O relatório nasce com status DRAFT (Rascunho) e pode ser editado livremente pela Organização ou pelo Investidor nesse estágio. reports.service.ts → saveDraft()
3
A Organização pode editar relatórios em DRAFT e RETURNED. O Investidor pode editar apenas em DRAFT. reports.service.ts → saveDraft()
4
Para enviar um relatório, é obrigatório ter pelo menos um comprovante de despesa (nota fiscal ou recibo) anexado. O sistema verifica tanto em tabela normalizada quanto em dados legados. reports.service.ts → submitReport()
5
Apenas relatórios em DRAFT ou RETURNED podem ser enviados para análise. Relatórios já enviados, aprovados ou em outro estado são bloqueados. reports.service.ts → submitReport()
6
Se o projeto possui um Consultor ativo (verificado na tabela project_consultants com active = true), é exclusivamente o Consultor quem analisa e aprova/devolve. O Investidor fica impedido de tomar essa ação. reports.service.ts → reviewReport()
7
Se o projeto não possui Consultor ativo, o Investidor assume diretamente a análise e aprovação. Consultores não vinculados ao projeto ficam impedidos. reports.service.ts → reviewReport()
8
Apenas um analista por vez é responsável pela aprovação: ou o Consultor, ou o Investidor — nunca ambos simultaneamente. O sistema valida isso a cada ação. reports.service.ts → projectHasActiveConsultant()
9
O Consultor deve ser o ativo no projeto específico para poder aprovar. Ter perfil de consultor não é suficiente — precisa estar vinculado e ativo naquele projeto. reports.service.ts → isActiveConsultantOnProject()
10
Ao devolver, o analista inclui um comentário obrigatório. Ao aprovar, o comentário é opcional. Ambos geram um registro na tabela report_reviews. ReviewReportButtons.tsx
11
Relatórios devolvidos ficam com status RETURNED. A Organização pode reabrir para edição e reenviar. O ciclo se repete sem limite. reports.service.ts → reopenToDraft()
12
O relatório aprovado fica permanentemente travado. Nenhum perfil pode alterar, editar ou reenviar um relatório com status APPROVED. lib/status.ts → REPORT_FLOW
13
Os dados financeiros são preenchidos dentro dos relatórios em tabelas normalizadas (report_financial_items, report_receipts, report_reallocations, report_bank_statements). report-financial.service.ts
14
A aba Financeiro do projeto exibe uma visão consolidada automática que agrega itens, comprovantes e extratos de todos os relatórios do projeto. report-financial.service.ts → getProjectFinancialAggregation()
15
O sistema mantém versionamento completo. Cada mudança de status gera uma nova versão na tabela report_versions, preservando o histórico completo do relatório. reports.service.ts → saveDraft(), submitReport()
16
A resolução de perfil segue prioridade INVESTOR > CONSULTANT > ORG. Se um usuário tem múltiplos vínculos, o perfil de maior prioridade prevalece. lib/roles.ts → getPrimaryRole()

Versão para Apresentação ao Cliente

Bruno,

Segue o resumo do fluxo final do Transparência Social, conforme definido e implementado no sistema.

O sistema funciona da seguinte forma: a Organização cria o relatório do projeto, preenche todos os dados — incluindo a parte financeira (itens de orçamento, gastos, remanejamentos, comprovantes de despesas e extratos bancários) — e envia para análise. Para enviar, é obrigatório ter ao menos um comprovante de despesa anexado.

A análise funciona de forma dinâmica, dependendo da configuração de cada projeto. Se existe um Consultor vinculado e ativo no projeto, é ele quem analisa e decide: aprova ou devolve para ajustes. Se não existe Consultor, o Investidor assume esse papel e faz a aprovação diretamente. Nunca os dois atuam ao mesmo tempo no mesmo projeto — o sistema garante que apenas um responsável por vez toma a decisão.

Se o relatório for devolvido, a Organização recebe de volta com os comentários do analista indicando o que precisa ser corrigido. Ela faz os ajustes e reenvia. Esse ciclo se repete quantas vezes for necessário até a aprovação.

Uma vez aprovado, o relatório fica travado permanentemente e todos os dados financeiros são consolidados automaticamente na aba Financeiro do projeto — dando visibilidade total e atualizada sobre a aplicação dos recursos.

Resumindo as permissões: a Organização é operacional (cria, preenche, envia, corrige após devolução). O Consultor, quando presente, é o analista exclusivo. O Investidor aprova diretamente apenas quando não há Consultor. Todos os perfis têm visibilidade sobre os dados financeiros consolidados.

Apêndice Técnico

Referência dos arquivos-chave do sistema e a lógica de negócio implementada em cada um. Baseado diretamente no código-fonte.

TS lib/roles.ts

Propósito: Define os 3 perfis do sistema e a resolução de prioridade.

getPrimaryRole(ctx) — resolve o perfil principal: INVESTOR > CONSULTANT > ORG
getRoleLabel(role) — retorna label amigável ("Empresa / Financiador", "Consultor", "Organização Social")
Tipo PrimaryRole = "INVESTOR" | "CONSULTANT" | "ORG"
TS lib/status.ts

Propósito: Define os status possíveis e as transições permitidas dos relatórios.

Status: DRAFT, SUBMITTED, RETURNED, APPROVED
Transições: DRAFT→SUBMITTED, SUBMITTED→APPROVED|RETURNED, RETURNED→SUBMITTED, APPROVED→(nenhuma)
Labels PT-BR: Rascunho, Enviado, Devolvido, Aprovado
TS services/reports.service.ts

Propósito: Arquivo central de toda a lógica de negócio dos relatórios. Controla permissões, validações e transições de status.

projectHasActiveConsultant(projectId) — consulta project_consultants onde active = true. Retorna boolean. Determina quem aprova.
isActiveConsultantOnProject(projectId, userId) — verifica se o usuário específico é o consultor ativo do projeto.
saveDraft(reportId, dataJson, userId) — Permite edição. Validações: (1) usuário é ORG ou INVESTOR; (2) ORG pode editar DRAFT e RETURNED; (3) INVESTOR pode editar apenas DRAFT. Cria nova versão ou atualiza existente.
submitReport(reportId, userId) — Envia para análise. Validações: (1) usuário é ORG ou INVESTOR; (2) status é DRAFT ou RETURNED; (3) pelo menos 1 comprovante existe (verifica tabela normalizada + JSON legado). Cria nova versão SUBMITTED e atualiza submitted_at.
reviewReport(reportId, comment, userId, decision) — Função central de aprovação/devolução. Validações: (1) usuário é INVESTOR ou CONSULTANT; (2) se projectHasActiveConsultant → só CONSULTANT pode agir, e deve ser o ativo no projeto; (3) se não há consultor → só INVESTOR pode agir. Insere registro em report_reviews e atualiza status.
reopenToDraft(reportId, userId) — Reabre para edição. ORG pode reabrir RETURNED. INVESTOR pode reabrir qualquer status exceto DRAFT.
approveReport() e returnReport() — wrappers que chamam reviewReport() com a decisão correspondente.
getLatestReview(reportId) — busca a revisão mais recente para exibir comentário do analista.
TS services/report-financial.service.ts

Propósito: CRUD das seções financeiras do relatório (seções 12-15) e consolidação no nível do projeto.

Tabelas: report_financial_items, report_financial_summary, report_reallocations, report_receipts, report_bank_statements
listFinancialItems(), upsertFinancialItem(), deleteFinancialItem() — CRUD itens financeiros (seção 12)
getFinancialSummary(), upsertFinancialSummary() — resumo financeiro por relatório
listReallocations(), upsertReallocation(), deleteReallocation() — remanejamentos (seção 13)
listReceipts(), upsertReceipt(), deleteReceipt() — comprovantes com upload (seção 14)
listBankStatements(), createBankStatement(), deleteBankStatement() — extratos (seção 15)
getProjectFinancialAggregation(projectId) — Consolidação. Busca todos os relatórios do projeto e agrega em paralelo: itens financeiros, comprovantes e extratos. Retorna totais: total_budget_planned, total_spent, total_receipts_value, total_bank_statements, total_items.
TS app/actions/report.actions.ts

Propósito: Server actions (Next.js) que conectam a UI às funções do service.

createReportAction() — verifica se role é ORG ou INVESTOR antes de criar. Busca template ativo para o tipo do projeto.
submitReportAction() → chama submitReport()
approveReportAction() → chama approveReport()
returnReportAction() → chama returnReport()
reopenReportToDraftAction() → chama reopenToDraft()
TSX app/dashboard/reports/[id]/page.tsx

Propósito: Página de detalhe do relatório. Calcula as permissões e renderiza botões condicionais.

canEdit = (ORG e status DRAFT|RETURNED) ou (INVESTOR e status DRAFT)
canSubmit = (ORG e status DRAFT|RETURNED) ou (INVESTOR e status DRAFT)
canReopen = ORG e status RETURNED
canReview = status SUBMITTED e [(hasConsultant e isConsultant) ou (!hasConsultant e isInvestor)]
Chama projectHasActiveConsultant() para determinar quem pode avaliar
TSX components/dashboard/ReviewReportButtons.tsx

Propósito: Componente de botões de avaliação (Aprovar / Devolver p/ ajustes).

Dois botões: "Aprovar" (verde) e "Devolver p/ ajustes" (âmbar)
Modal com campo de comentário. Ao devolver, comentário é obrigatório. Ao aprovar, é opcional.
Chama approveReportAction() ou returnReportAction()
Recebe prop roleLabel para exibir "consultor" ou "financiador" no texto do modal
TSX components/projects/ProjectFinancial.tsx

Propósito: Componente da aba Financeiro do projeto. Exibe visão consolidada.

Recebe projectId, role, financialData (retorno de getProjectFinancialAggregation())
Exibe KPIs: total de itens, orçamento planejado, gastos realizados, quantidade de comprovantes
Tabela consolidada de itens financeiros de todos os relatórios
Tabela consolidada de comprovantes
Lista de extratos bancários
Compatibilidade com dados legados: exibe dados JSON antigos quando tabelas normalizadas estão vazias
